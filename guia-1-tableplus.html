⚠️ Método 1 — Con TablePlus

Deploy de App Web
usando TablePlus

En este método cargamos el schema de la base de datos manualmente usando TablePlus. Es útil cuando el backend no tiene migración automática.

Railway MySQL TablePlus Vercel Node.js React.js
0
Introducción

Resumen del proceso

Antes de empezar, entendé qué vas a hacer y en qué orden.

🗺️ Flujo completo del deploy

  • Crear MySQL en Railway
    Railway nos da una base de datos MySQL lista para usar con credenciales automáticas.
  • Conectar TablePlus a Railway
    Usamos TablePlus como cliente visual para conectarnos a MySQL y ejecutar el schema.
  • Ejecutar el schema.sql
    Creamos todas las tablas en la base de datos ejecutando cada CREATE TABLE por separado.
  • Hacer el cambio en api.js
    Cambiamos la URL del backend de hardcodeada a variable de entorno en el frontend.
  • Deploy del backend en Railway
    Conectamos el repo de GitHub, configuramos el Root Directory y las variables de entorno.
  • Deploy del frontend en Vercel
    Conectamos el repo, configuramos la carpeta del frontend y la URL del backend.
ℹ️
La arquitectura final es: Vercel (frontend React) → Railway Backend (Node.js/Express) → Railway MySQL. Los tres se comunican entre sí en producción.
0
Antes de empezar

Prerrequisitos

Asegurate de tener todo esto antes de empezar.

  • Cuenta en GitHub con el repositorio del proyecto
    El repositorio debe ser tuyo (no un fork o repo ajeno sin acceso). Railway y Vercel necesitan permisos para leer el repo.
  • TablePlus instalado
    Descargalo gratis desde tableplus.com. La versión gratuita permite hasta 2 conexiones abiertas simultáneas, lo cual es suficiente.
  • El archivo schema.sql del proyecto
    Está en la carpeta backend/database/schema.sql del repositorio. Lo necesitás para crear las tablas en MySQL.
  • Cambio en api.js hecho y subido a GitHub
    Ver sección "Cambio en api.js" más adelante. Este cambio debe estar en el repo antes de hacer el deploy del frontend.
1
Base de Datos

Crear MySQL en Railway

Railway nos provee una base de datos MySQL completamente configurada en menos de un minuto.

  • Entrar a railway.app
    Ir a railway.app → clic en "Login" → "Login with GitHub" → autorizar Railway en tu cuenta de GitHub.
  • Crear un nuevo proyecto
    Clic en "New Project" → seleccionar "Add a service" → seleccionar "Database" → seleccionar "MySQL".
  • Esperar que el servicio quede activo
    Railway tarda aproximadamente 30-60 segundos. El estado debe cambiar de "Deploying" a "Active" con un punto verde.
  • Guardar las credenciales
    Clic en el servicio MySQL → pestaña "Variables". Vas a ver las siguientes variables generadas automáticamente por Railway. Guárdalas, las necesitarás después:

Cliente MySQL

Conectar TablePlus a Railway

TablePlus es una herramienta visual para gestionar bases de datos. La usamos para ejecutar el schema.

  • Abrir TablePlus
    Si no lo tenés instalado, descargarlo gratis desde tableplus.com e instalarlo.
  • Crear nueva conexión
    Clic en "Create a new connection" (o el ícono + abajo a la izquierda) → seleccionar "MySQL".
  • Llenar los datos de conexión
    Tomá los valores de la MYSQL_PUBLIC_URL de Railway. El formato es mysql://usuario:contraseña@host:puerto/database. Llenás cada campo así:

🔌 Datos de conexión para TablePlus

  • Probar la conexión
    Clic en "Test" en la parte inferior del formulario. Debe aparecer un mensaje en verde indicando que la conexión fue exitosa.
  • Conectar
    Clic en "Connect". Se abre la interfaz principal de TablePlus mostrando la base de datos railway (vacía por ahora).
🚫
Error "Lost connection": Significa que el host o puerto son incorrectos, o que Railway aún no terminó de iniciar el servicio MySQL. Esperá 1-2 minutos y volvé a intentar. Verificá que estés usando el puerto de la URL pública, no el 3306.
3
Tablas

Ejecutar el schema en TablePlus

Creamos todas las tablas ejecutando el schema SQL. Hay que hacerlo en partes porque TablePlus puede fallar con múltiples queries a la vez.

⚠️
Importante: El archivo schema.sql original tiene las líneas CREATE DATABASE profesor_ratings y USE profesor_ratings al inicio. Hay que ignorar o borrar esas líneas antes de ejecutar, porque Railway ya tiene su propia base de datos llamada railway.
  • Abrir el editor SQL
    En TablePlus presionar Ctrl+Q (Windows) o Cmd+Q (Mac) para abrir el editor de queries SQL.
  • Ejecutar cada CREATE TABLE por separado
    Copiar y pegar un solo CREATE TABLE a la vez en el editor y presionar "Run Current". No pegar todo el schema de una sola vez.
  • Orden correcto de las tablas
    Respetar el orden porque algunas tablas tienen llaves foráneas que dependen de otras:

📋 Orden de ejecución de tablas

  1. usuarios
  2. catedraticos
  3. cursos
  4. curso_catedratico (depende de cursos y catedraticos)
  5. publicaciones (depende de usuarios)
  6. comentarios (depende de publicaciones y usuarios)
  7. cursos_aprobados (depende de usuarios y cursos)
  • Verificar que se crearon las tablas
    En el panel izquierdo de TablePlus presionar Ctrl+R para refrescar. Deberían aparecer las 7 tablas listadas.
  • Ejecutar los datos de ejemplo (opcional)
    Si el proyecto tiene un archivo seed.sql, ejecutar también los INSERT para tener datos de prueba. Hacerlo después de crear todas las tablas.
🚫
Error "Failed to open referenced table": Significa que estás intentando crear una tabla que referencia otra que aún no existe. Revisá el orden de ejecución y asegurate de crear primero las tablas independientes.
4
Código

Cambio obligatorio en api.js

Este cambio hay que hacerlo antes de subir el frontend a Vercel. Sin este cambio, el frontend seguirá intentando conectarse a localhost.

🚨
Este es el error más común. Si olvidás este cambio, el frontend va a funcionar visualmente pero todas las peticiones al backend van a fallar con error ERR_CONNECTION_REFUSED.

Abrir el archivo frontend/src/services/api.js en VS Code y hacer este cambio:

frontend/src/services/api.js JavaScript 
// ❌ ANTES — URL hardcodeada, solo funciona en tu compu
const API_URL = 'http://localhost:5000/api';

// ✅ DESPUÉS — Lee variable de entorno, funciona en producción
const API_URL = process.env.REACT_APP_API_URL || 'http://localhost:5000/api';

💡 ¿Por qué este cambio?

process.env.REACT_APP_API_URL lee una variable de entorno que vamos a configurar en Vercel con la URL real del backend de Railway. El || significa "o" — si la variable no existe (como cuando trabajamos en local), usa localhost como fallback. Así el mismo código funciona en desarrollo y producción sin cambios adicionales.

React solo expone al navegador las variables que empiezan con REACT_APP_. Si la variable no tiene ese prefijo, React no la incluye en el build por seguridad.

  • Hacer el cambio en VS Code
    Abrir frontend/src/services/api.js, reemplazar la línea de API_URL y guardar el archivo.
  • Hacer commit y push a GitHub
    Con GitKraken: escribir un mensaje de commit como "fix: usar variable de entorno para API URL" → Stage All → Commit → Push. Con terminal: git add . && git commit -m "fix: api url" && git push.
5
Servidor

Deploy del backend en Railway

Conectamos el repositorio de GitHub al proyecto de Railway para que haga el deploy del backend automáticamente.

⚠️
Crítico: El servicio del backend debe estar en el mismo proyecto de Railway que el servicio MySQL. Si están en proyectos diferentes, no se pueden comunicar internamente y el backend no podrá conectarse a la BD.
  • Agregar servicio en el mismo proyecto
    En Railway, dentro del mismo proyecto donde está MySQL, clic en el botón "+" o "Add a service" → seleccionar "GitHub Repo".
  • Buscar y seleccionar el repositorio
    Buscar el nombre de tu repositorio. Si no aparece, clic en "Configure GitHub App", autorizar acceso al repo específico en GitHub, y volver a Railway.
  • Railway intenta hacer deploy automáticamente
    Va a fallar porque aún no hemos configurado el Root Directory ni las variables. Ese error es esperado en este punto.
6
Configuración

Configurar el Root Directory

El Root Directory le dice a Railway en qué carpeta del repositorio está el backend.

  • Ir a Settings del servicio del backend
    Clic en el servicio del backend (no en MySQL) → pestaña "Settings".
  • Configurar el Root Directory
    Buscar la sección "Root Directory" → clic en el campo → escribir la ruta según la estructura de tu repositorio. Ejemplo:
Root Directory en Railway Path 
# Si tu repo tiene esta estructura:
# mi-repo/
#   ├── profesor-ratings-app/
#   │   ├── backend/
#   │   └── frontend/

profesor-ratings-app/backend

# Si el backend está directo en la raíz del repo:
backend
💡
Para saber la ruta correcta, abrí el repositorio en GitHub y fijate cómo está organizado. La ruta en Root Directory debe llegar exactamente hasta la carpeta que contiene el package.json del backend.

Entorno

Variables de entorno del backend

Configuramos las variables que reemplazan el archivo .env en producción.

  • Ir a Variables del servicio del backend
    Clic en el servicio del backend → pestaña "Variables".
  • Agregar las variables de MySQL usando referencias
    Para las variables de BD, en lugar de copiar y pegar la contraseña, usar "Add Reference" → clic en el ícono {} al lado del campo de valor → buscar y seleccionar la variable del servicio MySQL. Esto crea una referencia dinámica.
  • Agregar el resto de variables manualmente
    Las que no son de MySQL se escriben directamente en el campo de valor.

8
URL pública

Generar el dominio del backend

Railway nos da una URL pública para que el frontend pueda comunicarse con el backend.

  • Ir a Settings → Networking
    En el servicio del backend → Settings → sección "Networking" → "Public Networking".
  • Clic en "Generate Domain"
    Railway genera una URL del tipo tu-app.up.railway.app. Copiá esa URL.
  • Verificar que el backend responde
    Abrir en el navegador: https://tu-dominio.up.railway.app/api/health. Debe responder con un JSON como {"message":"Servidor funcionando correctamente"}.
✅
Si el endpoint /api/health responde correctamente, el backend está funcionando y conectado a la base de datos. Si da error 500, revisá los logs en Railway → Deployments → View Logs.
9
Frontend

Deploy del frontend en Vercel

Vercel está especializado en frontends React y hace el build y deploy automáticamente.

  • Crear cuenta en Vercel
    Ir a vercel.com → "Sign Up" → "Continue with GitHub" → autorizar Vercel en tu cuenta de GitHub.
  • Importar el proyecto
    Clic en "Add New Project" → "Import Git Repository" → buscar y seleccionar tu repositorio de GitHub.
  • Configurar el Root Directory
    En la pantalla de configuración, buscar "Root Directory" → clic en "Edit" → escribir la ruta a la carpeta del frontend. Ejemplo: profesor-ratings-app/frontend.
  • Vercel detecta React automáticamente
    Vercel detecta que es una app React y configura automáticamente el comando de build (npm run build) y la carpeta de salida (build/). No hay que cambiar nada.
10
Variables

Configurar variables en Vercel

Antes de hacer deploy, agregar la variable de entorno con la URL del backend.

  • Expandir "Environment Variables"
    En la pantalla de configuración del deploy en Vercel, buscar la sección "Environment Variables" y expandirla.
  • Agregar la variable REACT_APP_API_URL
    En el campo Name escribir: REACT_APP_API_URL. En el campo Value pegar la URL del backend de Railway seguida de /api.
Variable en Vercel ENV 
REACT_APP_API_URL = https://tu-backend.up.railway.app/api

# Importante: incluir /api al final
# Reemplazar "tu-backend" con el dominio que generó Railway
  • Clic en "Deploy"
    Vercel construye la app React con npm run build y la despliega. El proceso tarda entre 1 y 3 minutos.
  • Verificar que el deploy dice "Ready"
    Cuando el estado cambia a "Ready", clic en el dominio generado para abrir la app en el navegador.
  • Actualizar CORS_ORIGIN en Railway
    Copiar la URL de Vercel → ir a Railway → servicio backend → Variables → editar CORS_ORIGIN → pegar la URL de Vercel → Railway hace redeploy automático.
⚠️
Si cambiás variables en Vercel después del deploy, hay que hacer Redeploy para que tomen efecto. Ir a Deployments → clic en los tres puntos del último deploy → Redeploy.
11
Pruebas

Verificar que todo funciona

Checklist final para confirmar que el deploy está completo.

  • Backend responde en /api/health
    Abrir https://tu-backend.up.railway.app/api/health en el navegador. Debe devolver JSON con mensaje de éxito.
  • Frontend carga correctamente
    Abrir la URL de Vercel. Debe mostrar la pantalla de login de la aplicación.
  • Registro de usuario funciona
    Intentar registrar un nuevo usuario. Si funciona, el frontend está comunicándose correctamente con el backend y el backend con la BD.
  • Login funciona
    Iniciar sesión con el usuario recién creado. Si funciona, el JWT y la autenticación están bien configurados.
  • Revisar consola del navegador
    Presionar F12 → pestaña Console. No debe haber errores de CORS ni de conexión rechazada. Si los hay, revisar la sección de errores comunes.
12
Troubleshooting

Errores comunes y soluciones

❌ ERR_CONNECTION_REFUSED — localhost:5000

Causa: El archivo api.js del frontend tiene la URL hardcodeada como localhost y no se hizo el cambio a variable de entorno.
Solución: Hacer el cambio en api.js, commit, push y redeploy en Vercel.

❌ CORS Policy Error

Causa: La variable CORS_ORIGIN en el backend no tiene la URL correcta de Vercel, o tiene la URL vieja.
Solución: Actualizar CORS_ORIGIN en Railway con la URL exacta de Vercel y esperar el redeploy.

❌ Error 500 — Table doesn't exist

Causa: Las tablas no fueron creadas en la base de datos de Railway.
Solución: Reconectarse con TablePlus y ejecutar el schema nuevamente. Verificar que las tablas aparezcan en el panel izquierdo.

❌ ENOTFOUND MYSQLHOST

Causa: La variable DB_HOST tiene el texto literal "MYSQLHOST" en lugar de una referencia al servicio MySQL de Railway.
Solución: En Railway → Variables → editar DB_HOST → usar el ícono {} → seleccionar ${{MySQL.MYSQLHOST}}.

❌ Lost connection to MySQL en TablePlus

Causa: Puerto o contraseña incorrectos, o el servicio MySQL aún está iniciando.
Solución: Verificar el puerto en MYSQL_PUBLIC_URL (no es el 3306 por defecto). Esperar 2 minutos si el servicio es nuevo. Probar cambiando SSL mode a DISABLED.

❌ Error creating build plan with Railpack

Causa: El Root Directory no está configurado o apunta a una carpeta que no tiene package.json.
Solución: Verificar la estructura del repo en GitHub y configurar correctamente el Root Directory en Settings del servicio.

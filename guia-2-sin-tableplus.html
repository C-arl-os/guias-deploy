Introducción

¿En qué se diferencia de la Guía 1?

La diferencia principal está en cómo se crean las tablas en la base de datos.

✨ La ventaja de este método

En lugar de conectarnos manualmente con TablePlus para ejecutar el schema, agregamos un script de migración al backend. Este script corre automáticamente cada vez que el servidor inicia y crea las tablas si no existen. Así no dependemos de herramientas externas ni pasos manuales.

Antes de empezar

Prerrequisitos

Lo único que necesitás es el repositorio en GitHub. No necesitás instalar nada adicional.

  • Cuenta en GitHub con el repositorio del proyecto
    El repositorio debe ser tuyo con permisos de escritura. Railway y Vercel necesitan acceso para leer el código.
  • VS Code (o cualquier editor de código)
    Necesitás editar tres archivos del proyecto antes del deploy.
  • Git instalado (GitKraken o terminal)
    Para hacer commit y push de los cambios al repositorio de GitHub.
1
Código nuevo

Crear el archivo migrate.js

Este archivo contiene el script que crea todas las tablas automáticamente. Es el núcleo de este método.

  • Crear el archivo en la ubicación correcta
    En VS Code, crear el archivo en: backend/src/config/migrate.js. Si la carpeta config no existe, crearla primero.
  • Pegar el siguiente contenido completo
    Copiar y pegar exactamente este código en el archivo:
backend/src/config/migrate.js JavaScript 
const pool = require('./database');

const migrate = async () => {
  const queries = [
    // Tabla de usuarios
    `CREATE TABLE IF NOT EXISTS usuarios (
      id INT PRIMARY KEY AUTO_INCREMENT,
      registro_academico VARCHAR(10) UNIQUE NOT NULL,
      nombres VARCHAR(100) NOT NULL,
      apellidos VARCHAR(100) NOT NULL,
      email VARCHAR(100) UNIQUE NOT NULL,
      password_hash VARCHAR(255) NOT NULL,
      fecha_creacion TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
      fecha_actualizacion TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
    )`,
    // Tabla de catedráticos
    `CREATE TABLE IF NOT EXISTS catedraticos (
      id INT PRIMARY KEY AUTO_INCREMENT,
      nombre VARCHAR(150) NOT NULL,
      apellido VARCHAR(150) NOT NULL,
      email VARCHAR(100),
      fecha_creacion TIMESTAMP DEFAULT CURRENT_TIMESTAMP
    )`,
    // Tabla de cursos
    `CREATE TABLE IF NOT EXISTS cursos (
      id INT PRIMARY KEY AUTO_INCREMENT,
      codigo VARCHAR(20) UNIQUE NOT NULL,
      nombre VARCHAR(150) NOT NULL,
      descripcion TEXT,
      creditos INT,
      fecha_creacion TIMESTAMP DEFAULT CURRENT_TIMESTAMP
    )`,
    // Relación cursos-catedráticos
    `CREATE TABLE IF NOT EXISTS curso_catedratico (
      id INT PRIMARY KEY AUTO_INCREMENT,
      curso_id INT NOT NULL,
      catedratico_id INT NOT NULL,
      semestre VARCHAR(20),
      anno INT,
      FOREIGN KEY (curso_id) REFERENCES cursos(id),
      FOREIGN KEY (catedratico_id) REFERENCES catedraticos(id)
    )`,
    // Tabla de publicaciones
    `CREATE TABLE IF NOT EXISTS publicaciones (
      id INT PRIMARY KEY AUTO_INCREMENT,
      usuario_id INT NOT NULL,
      tipo ENUM('catedratico', 'curso') NOT NULL,
      referencia_id INT NOT NULL,

      id INT PRIMARY KEY AUTO_INCREMENT,
      usuario_id INT NOT NULL,
      curso_id INT NOT NULL,
      calificacion DECIMAL(5,2),

module.exports = migrate;

💡 ¿Cómo funciona este script?

El script define un array de queries SQL y las ejecuta una por una en orden. Usamos CREATE TABLE IF NOT EXISTS para que no falle si las tablas ya existen — el script puede correr múltiples veces sin problema. Para los INSERT usamos INSERT IGNORE o condiciones WHERE NOT EXISTS para no duplicar datos.

2
Modificar

Modificar server.js

Hay que llamar al script de migración antes de levantar el servidor, para que las tablas estén listas cuando lleguen las primeras peticiones.

  • Abrir backend/server.js en VS Code
    Este archivo es el punto de entrada del backend — es el que corre cuando ejecutamos npm start.
  • Reemplazar el contenido completo con este código
    El cambio consiste en importar migrate.js y ejecutarlo antes de que el servidor empiece a escuchar peticiones:
backend/server.js — ANTES ❌ JavaScript 
const app = require('./src/app');
require('dotenv').config();

const PORT = process.env.PORT || 5000;

app.listen(PORT, () => {
  console.log(`Servidor corriendo en puerto ${PORT}`);
});
backend/server.js — DESPUÉS ✅ JavaScript 
const app = require('./src/app');
const migrate = require('./src/config/migrate'); // ← LÍNEA NUEVA
require('dotenv').config();

const PORT = process.env.PORT || 5000;

// Primero crea las tablas, luego inicia el servidor
migrate()                                           // ← LÍNEA NUEVA
  .then(() => {                                       // ← LÍNEA NUEVA
    app.listen(PORT, () => {
      console.log(`Servidor corriendo en puerto ${PORT}`);
    });
  })                                                  // ← LÍNEA NUEVA
  .catch((err) => {                                   // ← LÍNEA NUEVA
    console.error('Error en migración:', err);       // ← LÍNEA NUEVA
    process.exit(1);                                 // ← LÍNEA NUEVA
  });                                                 // ← LÍNEA NUEVA

💡 ¿Por qué este orden?

Si el servidor empezara a aceptar peticiones antes de que las tablas estén creadas, cualquier petición que intente leer o escribir en la BD daría error. Al usar .then(), garantizamos que la migración termine exitosamente antes de abrir el servidor. Si la migración falla, .catch() detiene el proceso con process.exit(1) para que Railway sepa que algo falló.

Código frontend

Cambio en api.js del frontend

Este cambio es igual que en la Guía 1 — es obligatorio para que el frontend se conecte al backend de Railway en lugar de localhost.

🚨
No olvidar este cambio. Sin él, el frontend intentará conectarse a localhost:5000 en lugar de al backend de Railway y todas las peticiones fallarán.
frontend/src/services/api.js JavaScript 
// ❌ ANTES — URL hardcodeada, solo funciona en local
const API_URL = 'http://localhost:5000/api';

// ✅ DESPUÉS — Lee variable de entorno
const API_URL = process.env.REACT_APP_API_URL || 'http://localhost:5000/api';
  • Hacer el cambio en VS Code y guardar
    Abrir frontend/src/services/api.js, reemplazar la línea de API_URL y guardar.
  • Hacer commit y push de los TRES archivos
    Los tres cambios — migrate.js (nuevo), server.js (modificado) y api.js (modificado) — deben estar en el repositorio de GitHub antes de hacer el deploy. Con GitKraken: Stage All → mensaje de commit → Commit → Push.
✅
Una vez que los tres cambios estén en GitHub, el código está listo para el deploy. Railway va a detectar el migrate.js y lo ejecutará automáticamente al iniciar.

Base de Datos

Crear MySQL en Railway

El proceso es idéntico a la Guía 1. La diferencia es que no necesitamos usar TablePlus después.

  • Entrar a railway.app
    Ir a railway.app → "Login" → "Login with GitHub" → autorizar Railway.
  • Crear proyecto nuevo con MySQL
    Clic en "New Project" → "Add a service" → "Database" → "MySQL". Esperar que el servicio quede Active.
  • No es necesario hacer nada más con la BD
    A diferencia de la Guía 1, no necesitamos conectarnos con TablePlus ni ejecutar ningún script manualmente. El backend creará las tablas automáticamente cuando inicie por primera vez.

🎯 La gran ventaja de este método

En la Guía 1, si borrás el proyecto de Railway y creás uno nuevo, tenés que volver a conectarte con TablePlus y ejecutar el schema manualmente. Con este método, simplemente redeploy el backend y las tablas se crean solas. Mucho más simple para proyectos que se replican frecuentemente.

Servidor

Deploy del backend en Railway

Conectamos el repositorio al mismo proyecto donde está MySQL.

⚠️
Mismo proyecto: El backend debe agregarse dentro del mismo proyecto de Railway donde creaste MySQL. Si están en proyectos separados, no se pueden comunicar por la red interna y el backend no podrá conectarse a la BD.
  • Agregar servicio en el mismo proyecto
    En el proyecto de Railway donde está MySQL, clic en "+" → "Add a service" → "GitHub Repo".
  • Seleccionar el repositorio
    Buscar el nombre del repo. Si no aparece, clic en "Configure GitHub App" para darle acceso a Railway al repositorio específico.
  • El deploy inicial puede fallar
    Es normal — falta configurar el Root Directory y las variables. Lo hacemos a continuación.

Configurar el Root Directory

Le decimos a Railway en qué carpeta del repositorio está el código del backend.

  • Ir a Settings del servicio del backend
    Clic en el servicio del backend (no MySQL) → pestaña "Settings".
  • Configurar Root Directory
    Buscar "Root Directory" → escribir la ruta exacta según la estructura del repositorio. Para verificar la estructura correcta, abrir el repo en GitHub y ver cómo están organizadas las carpetas.

Entorno

Variables de entorno del backend

Configuramos las variables que el código usa en producción.

  • Ir a Variables del servicio del backend
    Clic en el servicio del backend → pestaña "Variables".
  • Método rápido — pegar como .env
    En la parte superior del panel de Variables hay un campo grande. Podés pegar directamente el contenido como si fuera un archivo .env y Railway parsea las variables automáticamente.
Variables para pegar en Railway ENV 
PORT=5000
NODE_ENV=production
JWT_SECRET=tu_clave_secreta_muy_larga_y_segura
JWT_EXPIRE=7d
CORS_ORIGIN=*
ℹ️
Poné CORS_ORIGIN=* por ahora. Después del deploy en Vercel, actualizás esta variable con la URL exacta de tu app.

Para las variables de MySQL, agregalas usando referencias:

VariableCómo agregarlaResultado
DB_HOSTÍcono {} → seleccionar MySQL.MYSQLHOSTmysql.railway.internal
DB_USERÍcono {} → seleccionar MySQL.MYSQLUSERroot
DB_PASSWORDÍcono {} → seleccionar MySQL.MYSQLPASSWORD[contraseña generada]
DB_NAMEÍcono {} → seleccionar MySQL.MYSQLDATABASErailway
DB_PORTÍcono {} → seleccionar MySQL.MYSQLPORT3306
⚠️
Referencias vs texto plano: Para las variables de MySQL es importante usar referencias (el ícono {}) en lugar de copiar y pegar los valores. Si pegás el texto plano "MYSQLHOST" como valor de DB_HOST, el backend recibirá literalmente la palabra "MYSQLHOST" y no podrá conectarse a la base de datos.

URL pública

Generar dominio y verificar

Le damos una URL pública al backend y verificamos que funciona.

  • Generar el dominio
    Ir a Settings → Networking → Public Networking → clic en "Generate Domain". Railway asigna una URL del tipo tu-app.up.railway.app.
  • Esperar que el deploy termine
    Railway va a redeploy automáticamente cuando detecte las variables. En los logs debería aparecer en orden: "Conectado a la base de datos MySQL" → "✅ Migración completada — tablas listas" → "Servidor corriendo en puerto 5000".
  • Verificar el endpoint de salud
    Abrir en el navegador: https://tu-dominio.up.railway.app/api/health. Debe responder con JSON indicando que el servidor está funcionando.
✅
Si en los logs de Railway ves las tres líneas en ese orden y el endpoint /api/health responde, el backend está completamente funcional con las tablas creadas. ¡No necesitaste abrir TablePlus en ningún momento!

Troubleshooting

Errores comunes y soluciones

❌ Error en migración: ECONNREFUSED

Causa: El backend y MySQL están en proyectos diferentes de Railway, o las variables de BD no están configuradas correctamente como referencias.
Solución: Verificar que ambos servicios estén en el mismo proyecto. Verificar que DB_HOST tenga el valor mysql.railway.internal (no el texto "MYSQLHOST").

❌ Cannot find module './src/config/migrate'

Causa: El archivo migrate.js no fue creado o no fue subido al repositorio de GitHub.
Solución: Verificar que el archivo existe en backend/src/config/migrate.js en GitHub. Si no está, hacer commit y push del archivo.

❌ ERR_CONNECTION_REFUSED — localhost:5000

Causa: El cambio en api.js no fue hecho o no fue subido a GitHub antes del deploy de Vercel.
Solución: Hacer el cambio, commit, push, y luego hacer Redeploy en Vercel.

❌ CORS Policy Error

Causa: CORS_ORIGIN en Railway tiene * (temporal) o una URL incorrecta.
Solución: Actualizar CORS_ORIGIN con la URL exacta de Vercel y esperar el redeploy automático.

❌ El servidor no inicia después de la migración

Causa: Hay un error de sintaxis en migrate.js o en server.js.
Solución: Revisar los logs completos en Railway → Deployments → View Logs para ver el error exacto. Corregir en VS Code, commit y push.
